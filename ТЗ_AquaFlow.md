# Техническое задание: AquaFlow

**Демо-проект: симулятор трубопроводной сети со SCADA-логикой + нейросетевой предиктор потока**

Версия 1.0 · Документ предназначен для исполнения агентом (Claude Code).

---

## 1. Обзор и цель

AquaFlow — десктопное приложение, которое имитирует распределительную сеть труб (в духе SCADA-систем) и умеет двумя способами определять, в какие точки приёма дойдёт вода при заданных настройках:

1. **Реальный прогон** — детерминированный симулятор физически «прогоняет» воду по трубам и точно показывает результат (это истина).
2. **Расчёт нейросетью** — обученная нейросеть за миллисекунду предсказывает результат, не запуская симуляцию.

Смысловой гвоздь демо: сравнение «предсказание нейросети vs реальный результат», с накоплением точности вживую. Это демонстрирует полный ML-конвейер (генерация данных → обучение → инференс → сравнение с реальностью) на понятной визуальной задаче.

**Целевая реакция аудитории:** «неожиданно — человек реально разобрался в ML, а не просто прикрутил чужое API».

---

## 2. Термины

- **Вход (Source)** — точка, откуда подаётся вода. В сети их три: A, B, C. За один прогон активен ровно один вход.
- **Приёмник (Receiver)** — точка приёма воды. Их три: X, Y, Z. За прогон вода может дойти до нескольких приёмников (multi-label).
- **Узел (Junction)** — распределительный элемент с клапаном. В сети их семь: J1…J7.
- **Клапан (Valve)** — состояние узла, 0 или 1 (закрыт/открыт). Задаётся пользователем мышью.
- **Конфигурация (Config)** — {выбранный вход} + {состояния 7 клапанов}. Полностью определяет результат.
- **Результат (Outcome)** — множество приёмников, до которых дошла вода.

## 3. Архитектура

### 3.1. Модель живёт в процессе приложения (in-process)

Нейросеть — это файл весов на диске. Приложение при старте один раз загружает его в память через `IWaterPredictor`, дальше каждый вызов `Predict(config)` — синхронный, мгновенный. Никакого отдельного сервиса, сети или gRPC. Для однопользовательского десктопа это осознанно выбранная простая архитектура.

### 3.2. Solution из трёх проектов

```
AquaFlow.sln
├── AquaFlow.Core        (class library)  — доменная модель + детерминированный симулятор
├── AquaFlow.Ml          (class library)  — генерация данных, обучение, IWaterPredictor, метрики
└── AquaFlow.App         (Avalonia app)   — UI, канвас, вкладки, связывание всего вместе
```

Зависимости строго в одну сторону: `App → Ml → Core`. `Core` ни от чего не зависит и не знает про UI и ML. Это делает симулятор переиспользуемым и тестируемым.

### 3.3. Поток данных

```
Симулятор (Core.Run)  ──► генерирует размеченный датасет  ──► Postgres (samples)
                                                                    │
                                                                    ▼
                                              обучение MLP (Ml) ──► файл модели (.bin)
                                                                    │
                          ┌─────────────────────────────────────────┘
                          ▼
App при старте грузит модель ──► Predict(config)  ──► сравнение с реальным прогоном
                                                     └─► история и метрики в Postgres (runs)
```

Ключевая идея: **симулятор — это одновременно и движок анимации, и генератор обучающих меток.** Размечать данные руками не нужно — симулятор сам себе учитель.

## 5. Доменная модель симуляции (проект Core)

### 5.1. Топология сети

Сеть — ориентированный ацикличный граф: 3 входа → 7 узлов (4 слоя) → 3 приёмника.

Рёбра от входов:
- `A → J1`, `B → J2`, `C → J3`

Узлы, их исходящие рёбра и семантика клапана:

| Узел | Тип | Исходящие | valve = 0 | valve = 1 |
|---|---|---|---|---|
| J1 | диверт | J4, J5 | → J4 | → J5 |
| J2 | сплиттер | J4, J5 | → J4 | → J4 **и** J5 |
| J3 | диверт | J5, J6 | → J5 | → J6 |
| J4 | диверт | J6, J7 | → J6 | → J7 |
| J5 | сплиттер | J6, J7 | → J7 | → J6 **и** J7 |
| J6 | диверт | X, Y | → X | → Y |
| J7 | диверт | Y, Z | → Y | → Z |

- **Диверт** направляет входящую воду ровно в одно из двух рёбер.
- **Сплиттер** при `valve = 1` пускает воду **сразу в оба** исходящих ребра — именно это порождает multi-label (вода доходит до нескольких приёмников).

Глубина маршрута — до 4 звеньев; узлы связаны между собой (выход одного питает вход другого), поэтому результат зависит от комбинации клапанов, а не от каждого по отдельности. Это то, что нейросети предстоит выучить.

> Агент может уточнить/подстроить конкретные рёбра, но обязан сохранить свойства: граф **ацикличный**, детерминированный, есть минимум 2 сплиттера, глубина до 3–4 звеньев, все приёмники достижимы хотя бы при какой-то конфигурации.

### 5.2. API симулятора

```csharp
public sealed record SimulationResult(
    IReadOnlySet<Receiver> ReachedReceivers,   // множество приёмников с водой
    IReadOnlyList<Edge> TraversedEdges);       // рёбра, по которым прошла вода (для анимации)

public interface IPipeSimulator
{
    SimulationResult Run(SimConfig config);
}
```

`Run` реализуется как обход графа (BFS/DFS) от выбранного входа по «открытым» (согласно клапанам) рёбрам. Приёмник попадает в `ReachedReceivers`, если до него есть открытый путь. Полностью детерминирован: одинаковый `config` → одинаковый результат.

### 5.3. Обязательные тесты (xUnit)

Не менее 8 кейсов с вручную посчитанным ожидаемым результатом, включая: одиночный путь до одного приёмника; конфигурацию со сплиттером, дающую 2+ приёмника; вход, из которого при всех клапанах = 0 вода идёт по «нижнему» маршруту; крайние конфигурации (все 0, все 1).

## 6. Модель машинного обучения (проект Ml)

### 6.1. Постановка задачи

**Multi-label классификация.** Вход → вектор признаков; выход → 3 независимые бинарные метки (дошла ли вода до X, до Y, до Z).

- **Признаки (10 чисел):** one-hot входа `[isA, isB, isC]` + состояния клапанов `[J1..J7]`.
- **Метки (3 числа):** `[reachedX, reachedY, reachedZ]`, каждая 0/1.

### 6.2. Архитектура сети (MLP на TorchSharp)

```
Вход: 10 признаков
 └─ Linear(10 → 32) → ReLU
     └─ Linear(32 → 16) → ReLU
         └─ Linear(16 → 3) → Sigmoid   (3 вероятности, по одной на приёмник)
```

- Функция потерь: `BCELoss` (binary cross-entropy) по каждому выходу.
- Оптимизатор: Adam, lr ≈ 1e-3.
- Эпохи: ~200–500 (до выхода на плато), батч ~64.
- Порог классификации: вероятность ≥ 0.5 → «вода дошла».

Всё это прячется за интерфейсом, чтобы реализацию можно было заменить:

```csharp
public sealed record Prediction(
    IReadOnlyDictionary<Receiver, float> Probabilities,  // сырые вероятности 0..1
    IReadOnlySet<Receiver> PredictedReceivers);          // после порога 0.5

public interface IWaterPredictor
{
    Prediction Predict(SimConfig config);
}
```

### 6.3. Обучение

1. Сгенерировать датасет: пройти по всем 3 × 2⁷ = 384 конфигурациям (или случайной выборке, если сеть расширят) и для каждой вызвать `IPipeSimulator.Run` → пары (признаки → метки).
2. Сохранить датасет в Postgres (таблица `samples`).
3. Разбить на train/test (например 70/30) с фиксированным seed. **Тест держим отложенным** — на нём показываем, что сеть обобщает, а не заучивает таблицу.
4. Обучить MLP, сохранить веса в файл (`models/water_mlp.bin`).
5. Записать метаданные в Postgres (таблица `model_meta`).

### 6.4. Метрики (для вкладки «Метрики»)

- **Per-label accuracy** — точность отдельно по X, Y, Z.
- **Subset accuracy** — доля конфигураций, где все три метки предсказаны верно одновременно.
- **Матрица ошибок** по каждому приёмнику: TP / FP / FN / TN.
- **Уверенность** — сырые вероятности сигмоиды для текущей конфигурации.

Целевой ориентир качества на отложенном тесте: subset accuracy ≥ 0.95. Если ниже — увеличить число эпох / ширину скрытого слоя / объём train.

## 7. База данных (PostgreSQL)

Поднимается через `docker-compose.yml` в корне репозитория. Схема и seed — в виде миграций/SQL в репозитории; сами данные в репозиторий не коммитятся.

Таблицы:

- **samples** — сгенерированный обучающий датасет.
  `id, source (text), valves (jsonb), reached_receivers (text[])`
- **runs** — история прогонов из приложения.
  `id, created_at, mode (text: 'real'|'predict'), source (text), valves (jsonb), predicted_receivers (text[]), predicted_probs (jsonb), actual_receivers (text[]), was_correct (bool)`
- **model_meta** — реестр обученных моделей.
  `id, trained_at, epochs, subset_accuracy (float), file_path (text)`

## 8. Пользовательский интерфейс (проект App)

Три вкладки.

### 8.1. Вкладка «Симуляция»

- **Канвас** с визуализацией сети: 3 входа слева, 7 узлов в центре (по слоям), 3 приёмника справа, трубы — линиями между ними.
- **Выбор входа** — клик по одному из A/B/C (активный подсвечивается).
- **Клапаны** — клик по узлу переключает 0/1; текущее состояние видно на узле (например стрелка ↑/↓ или разный цвет).
- **Кнопка «Реальный прогон»** — запускает `IPipeSimulator.Run`, анимирует движение воды по `TraversedEdges`, **закрашивает синим** трубы и приёмники, куда дошла вода, показывает итог.
- **Кнопка «Расчёт (нейросеть)»** — вызывает `IWaterPredictor.Predict`, подсвечивает предсказанные приёмники (например синей рамкой/заливкой), показывает вероятности.

### 8.2. Главный вау-момент: «нейросеть против реальности»

Сценарий пользователя:
1. Выставляет вход и клапаны.
2. Жмёт «Расчёт» → сеть говорит «предсказываю: вода дойдёт до X и Z» (с вероятностями).
3. Жмёт «Реальный прогон» → вода реально течёт, приёмники заливаются синим.
4. Приложение сравнивает предсказание с фактом → «Совпало ✓» / «Ошибка ✗», и обновляет **счётчик точности вживую** (например «модель угадала 47 из 48 прогонов, 97.9%»).
5. Каждый прогон пишется в Postgres (`runs`), счётчик переживает перезапуск.

### 8.3. Вкладка «Метрики»

Мини-дашборд: subset accuracy и per-label accuracy, матрица ошибок по каждому приёмнику, столбики уверенности для последней конфигурации. Данные берутся из результатов на отложенном тесте + агрегатов из `runs`.

### 8.4. Вкладка «История»

Список прошлых прогонов из таблицы `runs`: время, вход, клапаны, что предсказано, что реально, совпало или нет. Простая таблица с прокруткой.

## 9. Milestones

Агент выполняет по одному майлстоуну за раз. После каждого: `dotnet build` должен быть зелёным, тесты (где есть) — проходить, и только тогда переход к следующему. Коммит на каждый майлстоун.

### M0 — Каркас
- Solution из трёх проектов, `docker-compose.yml` с Postgres, `.gitignore`, `README.md` с командами запуска.
- **Готово, когда:** `dotnet build` зелёный; `docker compose up` поднимает Postgres; приложение запускается с пустым окном и тремя вкладками.

### M1 — Симулятор (Core)
- Доменная модель сети (входы, узлы, приёмники, рёбра), `IPipeSimulator.Run`, xUnit-тесты.
- **Готово, когда:** ≥ 8 тестов проходят на вручную проверенных конфигурациях; результат детерминирован.

### M2 — Канвас и реальный прогон (App)
- Отрисовка сети, выбор входа, переключение клапанов мышью, кнопка «Реальный прогон» с анимацией и синей заливкой достигнутых приёмников, окно итога.
- **Готово, когда:** ручной прогон в UI совпадает с результатом `Core.Run` для любой конфигурации.

### M3 — Датасет и Postgres (Ml + БД)
- Генерация полного датасета симулятором, запись в `samples`, воспроизводимое train/test-разбиение.
- **Готово, когда:** таблица `samples` заполнена 384 записями; разбиение стабильно при фиксированном seed.

### M4 — Нейросеть (Ml)
- MLP на TorchSharp за `IWaterPredictor`, обучение, сохранение весов, загрузка при старте, метрики на тесте.
- **Готово, когда:** модель обучается за десятки секунд на CPU; subset accuracy на отложенном тесте ≥ 0.95; `Predict` возвращает вероятности и метки.

### M5 — Вау-момент (App)
- Кнопка «Расчёт», сравнение предсказания с реальным прогоном, живой счётчик точности, запись прогонов в `runs`.
- **Готово, когда:** цикл «предсказал → прогнал → совпало/нет → счётчик обновился» работает; счётчик восстанавливается из Postgres после перезапуска.

### M6 — Дашборд и история (App)
- Вкладка «Метрики» (accuracy, матрица ошибок, уверенность) и вкладка «История» (прогоны из `runs`).
- **Готово, когда:** метрики соответствуют данным в БД; матрица ошибок считается корректно; история отображается.

### M7 — Полировка (stretch, если останется время)
- Плавные анимации потока, аккуратный стиль, экспорт модели в ONNX как альтернативная реализация `IWaterPredictor`, опциональный gRPC-хост модели «на будущее».
- **Готово, когда:** демо выглядит отточенным; при желании модель работает и через ONNX без изменений в UI.

## 10. Нефункциональные требования

- **Платформа:** macOS arm64 (Apple Silicon), без GPU.
- **Производительность:** `Predict` — мгновенный; обучение — десятки секунд; анимация плавная.
- **Комментарии в коде — на русском.** Названия идентификаторов (классы, методы, переменные) — на английском. Тексты интерфейса — на русском.
- **Чистота:** `Core` не зависит от UI и ML; вся работа с моделью — только через `IWaterPredictor`; вся работа с симулятором — только через `IPipeSimulator`.

## 12. Инструкции для агента

1. Двигайся строго по майлстоунам, по одному за раз. Не переходи дальше, пока текущий не собирается и не проходит проверки готовности.

## 13. Чек-лист «вау» (критерии готового демо)

- [ ] Видно живую сеть труб, клапаны переключаются мышью.
- [ ] Реальный прогон красиво анимирует поток и заливает приёмники синим.
- [ ] Нейросеть предсказывает результат заранее — и почти всегда угадывает.
- [ ] Счётчик точности растёт на глазах и переживает перезапуск.
- [ ] Есть дашборд с метриками и матрицей ошибок — выглядит профессионально.
- [ ] Всё работает локально на MacBook, без облака и без внешних API.
- [ ] Ты можешь объяснить каждый слой: симулятор → данные → обучение → инференс → сравнение.
